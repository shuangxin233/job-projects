# 项目学习大纲：Dify Deep Research 行业研究 Agent

## 1. 项目定位

本项目基于 Dify 工作流复现 Deep Research 能力，目标不是企业知识库问答，而是构建一个“行业研究 / 竞品分析 / 主题调研报告生成”系统。

核心能力包括：

- 用户输入一个研究主题。
- 系统先追问澄清问题，补全研究目标和边界。
- Agent 将主题拆解成多个研究子方向。
- 通过本地资料、维基百科、Google/Bing 等多源检索补充上下文。
- 多个 LLM 节点分别生成不同章节内容。
- 最终合成为结构化 Markdown 研究报告。

你学习这个项目时，重点不是只会导入 Dify DSL，而是理解“一个复杂 AI 应用如何拆成工作流、Prompt、检索、生成、部署和微调几个模块”。

## 3. 第一阶段：跑通项目

目标：先把项目运行起来，不急着改源码。

学习任务：

1. 安装并启动 Dify。
2. 在 Dify 中导入 `Deep Researcher On Dify .yml`。
3. 配置可用的大模型供应商，例如 DeepSeek、OpenAI-compatible、Gemini 或本地模型。
4. 配置必要工具，例如 Wikipedia 搜索、网络搜索 API、本地知识检索或可替代的数据源。
5. 输入一个简单主题测试完整流程。

推荐测试主题：

- “新能源汽车换电行业发展趋势”
- “AI 编程工具 Cursor、Claude Code、Codex 的竞品分析”
- “跨境电商独立站增长策略研究”

验收标准：

- 能导入工作流。
- 能正常触发追问。
- 能生成 3 到 5 个子主题。
- 能输出一份结构化 Markdown 报告。

## 4. 第二阶段：理解工作流主链路

目标：看懂 Dify 工作流为什么这样设计。

重点节点：

- `开始`：接收用户研究主题。
- `问题分解`：判断主题是否清晰，并生成澄清问题。
- `第一次回复&问题1`、`问题2`、`问题3`、`问题4`：多轮追问用户需求。
- `问题参数提取`：从用户回答中抽取研究约束。
- `最终回复`：确认研究主题。
- `sub主题提取`：生成 3 到 5 个研究子标题。
- `sub主题1分析` 等多个分析节点：分章节生成内容。
- `维基百科搜索`：工具调用节点，用于补充外部信息。
- `template-transform`：将多个章节拼接成最终 HTML/Markdown 报告。

你需要画出一张自己的流程图，建议结构：

`用户主题 -> 澄清追问 -> 主题修正 -> 子主题拆解 -> 检索增强 -> 多章节生成 -> 报告合成`

## 5. 第三阶段：Prompt 工程学习

目标：理解每个 LLM 节点的 Prompt 分工。

重点阅读：

- 主题澄清 Prompt：如何判断用户问题是否清楚。
- 子主题生成 Prompt：如何控制输出 3 到 5 个研究角度。
- 章节生成 Prompt：如何让不同章节风格统一。
- 开头和结尾 Prompt：如何承接正文并形成完整报告。
- 输出格式 Prompt：如何要求 Markdown、标题层级、语言一致性。

练习任务：

1. 将“通用研究报告”改成“行业研究报告”模板。
2. 增加固定章节：
   - 行业背景
   - 市场规模
   - 竞品格局
   - 用户痛点
   - 商业模式
   - 风险与机会
   - 结论与建议
3. 改造 Prompt，让报告必须包含表格和要点摘要。

## 6. 第四阶段：RAG 与检索增强学习

目标：把“检索”从普通搜索升级成可解释的 RAG 模块。

原项目已有能力：

- 本地知识库 + 网络搜索的混合检索思路。
- Wikipedia 搜索工具节点。
- 可替换 Google/Bing/API 等检索源。

你要补的能力：

1. 把搜索结果保存为临时研究资料。
2. 对资料做切分、清洗、去重和摘要。
3. 对关键资料做向量化召回。
4. 在报告中输出“引用来源”或“证据片段”。

建议改造成：

`搜索结果 -> 文本清洗 -> 向量入库 -> 子主题检索 -> 带引用生成章节`

这能让项目区别于普通“联网搜索 + 总结”。

## 7. 第五阶段：Agent 能力学习

目标：让系统体现 Agent 的任务规划与工具调用，而不只是流水线。

需要理解的 Agent 行为：

- 根据主题判断需要追问什么。
- 根据用户回答重新修正研究主题。
- 自动生成研究子任务。
- 针对不同子任务选择不同检索策略。
- 汇总多个子任务结果。

二次开发方向：

1. 增加“研究计划生成”节点。
2. 增加“信息缺口判断”节点。
3. 增加“是否继续检索”的条件分支。
4. 增加“报告质量自评”节点。
5. 增加“重新生成低质量章节”的循环。

求职表达时，要把这些说成“Agent 规划、工具调用、反思评估、迭代生成”。

## 8. 第六阶段：Docker 与工程化部署

目标：把它从一个 Dify 工作流变成完整项目。

建议项目结构：

```text
project3-deep-research-agent/
  dify/
    Deep Researcher On Dify .yml
  backend/
    FastAPI 服务，负责调用 Dify API、保存任务、管理报告
  frontend/
    报告任务页面、历史报告页面、导出按钮
  data/
    示例主题、测试输入、生成结果
  finetune/
    训练数据、LlamaFactory 配置、评估结果
  docker-compose.yml
  README.md
```

Docker Compose 至少包含：

- Dify 服务，或说明如何连接已有 Dify。
- 后端 FastAPI。
- 前端 Web。
- 数据库，例如 PostgreSQL。
- 可选向量库，例如 Qdrant、Milvus 或 Chroma。

验收标准：

- 一条命令启动后端和前端。
- 前端可以提交研究主题。
- 后端可以调用 Dify Workflow API。
- 报告可以保存、查看和导出。

## 9. 第七阶段：微调扩展

目标：把“微调”自然接入项目，而不是硬塞关键词。

不要一开始就微调大模型。先收集数据，再做小范围 LoRA/QLoRA。

适合微调的任务：

- 资料摘要：把搜索结果压缩为高质量研究摘要。
- 报告风格：让输出更像咨询报告、投研报告或商业分析报告。
- 章节分类：判断一段资料属于市场、竞品、风险、机会还是政策。
- 标题生成：把子主题标题变得更专业。

推荐流程：

1. 用原工作流生成 30 到 100 份报告。
2. 人工挑选或改写高质量段落。
3. 构造 Alpaca 或 ShareGPT 格式训练集。
4. 用 LlamaFactory 训练 LoRA。
5. 对比微调前后：
   - 摘要准确性
   - 结构完整性
   - 语气专业度
   - 幻觉率
6. 将微调模型通过 OpenAI-compatible API、Ollama、vLLM 或 Xinference 接入 Dify。

简历上可以写：

“基于 LlamaFactory 对行业研究摘要与报告风格进行 LoRA 微调，并将微调模型以 OpenAI-compatible API 方式接入 Dify 工作流。”
